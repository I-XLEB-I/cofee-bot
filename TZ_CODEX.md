# ТЗ для Codex: рефакторинг coffee-bot

## Контекст

Рабочий файл: `/Users/mark/coffee-bot/bot.py` (~12 700 строк, один модуль).
Бот работает на проде через `systemd` на `185.119.58.78`, polling, таймзона Europe/Moscow.
Backup-файлы не трогать. Бизнес-логика и сущности описаны в `PROJECT_CONTEXT.md` — читать перед началом работы.

**Цель рефакторинга.** Снизить риск регрессий при дальнейшей разработке: добавить страховку тестами, повысить устойчивость к временным ошибкам Google Sheets, вычистить хардкод и секреты, разбить монолит на домены.

**Общие правила.**
- Поведение бота для пользователя должно остаться неизменным. Любое изменение контракта (новые кнопки, изменённые экраны, другие строки в Sheets) — только с явного согласия владельца.
- Каждая фаза заканчивается рабочим ботом: `python3 -m py_compile bot.py` без ошибок, `ruff check .` без новых замечаний, прогон по основным сценариям (обслуживание, ревизия, отчёт за день) успешен.
- Между фазами — отдельный коммит и ручной smoke-test на проде.
- Никаких изменений в структуре Google-таблиц без согласования. Названия листов и заголовки строк — константы, уже закреплённые в коде.

## Фаза 1. Безопасность и чистка

Критерий готовности: добавление/удаление сотрудника делается без правки кода и деплоя.

## Фаза 2. Устойчивость к ошибкам Google Sheets

**Зачем.** Сейчас на временной ошибке API (429/5xx) пользователь видит экран «⏳ Google Sheets перегружен» и жмёт «Повторить» руками. Автоматические короткие повторы уберут половину таких случаев. Плюс `_BOOK_CACHE` сейчас небезопасен относительно параллельного доступа.

### 2.1 Декоратор с retry/backoff для вызовов gspread

Сделать:
- В отдельном модуле (или в `bot.py` пока без разбиения — в Фазе 4 переедет) реализовать декоратор `@sheets_retry(max_attempts=3, base_delay=0.5, max_delay=4.0, jitter=True)`.
- Ретраим только на `gspread.exceptions.APIError` с HTTP-кодами `429`, `500`, `502`, `503`, `504`. Остальные ошибки пробрасываем без повторов.
- Задержка — экспоненциальная (`base_delay * 2**(attempt-1)`) с джиттером (±25%).
- Применить декоратор к низкоуровневым функциям доступа к Sheets: `get_records_with_rows`, `append_row_and_get_index`, `add_service_row`, `update_service_row`, `add_travel_row`, `add_photo_row`, `update_photo_row`, `add_salary_task_row`, `add_rent_payment_row`, `add_rent_document_row`, `get_or_create_worksheet` (только части с сетевыми вызовами, не локальный кэш).
- В логи — `logger.warning(...)` на каждом ретрае с номером попытки и HTTP-кодом. На финальной неудаче — существующий экран «⏳ Google Sheets перегружен» остаётся как последняя линия обороны.
- Все `gspread`-вызовы по-прежнему идут через `run_blocking`/`asyncio.to_thread`.

Критерий готовности: при имитации `APIError(429)` в первом вызове последующий ретрай проходит успешно, пользователь не видит экран «перегружен».

### 2.2 Блокировка на `_BOOK_CACHE`

Сейчас: глобальный dict читается и пишется из разных потоков (`run_blocking` запускает в пуле).

Сделать:
- Обернуть все чтения/записи в `_BOOK_CACHE` (включая `_BOOK_CACHE["worksheets"]`) в `threading.Lock`.
- Лок создаётся один раз на модуль: `_BOOK_CACHE_LOCK = threading.Lock()`.
- `_ENSURED_WORKSHEET_GROUPS` тоже под тем же локом — он используется в `ensure_worksheet_group`.
- Не держать лок во время сетевого вызова `client.open_by_key` — иначе все запросы выстроятся в очередь. Паттерн: под локом проверить/прочитать кэш, отпустить, сделать сетевой вызов, снова взять лок и записать результат. Допустимо, что два потока параллельно сделают по одному лишнему `open_by_key` — это безопасно и случается редко.

### 2.3 Обновить `show_sheets_busy_notice`

После внедрения retry экран «перегружен» показывается только при окончательном провале. Тексту это соответствует — менять не нужно. Убедиться, что `retry_callback` по-прежнему работает и запускает повтор той же операции.

## Фаза 3. Тесты на pure-функции

**Зачем.** 12 700 строк без тестов. Перед разбиением монолита (Фаза 4) нужна страховка на самых рискованных участках — парсерах и расчётной логике. Это чистые функции, их легко вынести и покрыть без мокания Telegram и Sheets.

### 3.1 Вынести pure-функции в модуль `core/`

Создать пакет:

```
coffee-bot/
  bot.py
  core/
    __init__.py
    dates.py
    water.py
    service_rules.py
    revision_import.py
    rows.py
```

Функции к переносу (из `bot.py` — найти по имени, скопировать как есть, импортировать обратно в `bot.py` через `from core.dates import ...`):

- `core/dates.py` — парсинг даты `дд.мм`, правило «зимняя дата = прошлый год», запрет будущих дат и дат старше года. Если в коде сейчас это размазано по нескольким местам — собрать в одну функцию `parse_manual_date(text: str, today: date) -> date` с документированным контрактом.
- `core/water.py` — нормализация воды (`"2,5"` → `2.5`, защита от `"25"` вместо `"2,5"`), форматирование воды для вывода (`2.5` → `"2,5"`).
- `core/service_rules.py` — логика «через день»: была ли точка обслужена вчера / позавчера, попадает ли в «Нужно сегодня» / «Можно не обслуживать сегодня» / «Уже обслужены». Функция принимает точку, список дат обслуживания, сегодняшнюю дату — возвращает категорию.
- `core/revision_import.py` — парсер обоих форматов импорта ревизии (блоки по точкам и табличный через `/`), включая суммирование `Кофе исп` + `Кофе кит` в `Кофе`, поддержку `Салфетки сухие`, сбор предупреждений о конфликтах.
- `core/rows.py` — `build_service_row_values` и подобные билдеры строк для Sheets, сериализация/десериализация списка работников (`serialize_salary_workers`, `normalize_salary_workers`).

Каждая функция — полностью чистая: без обращений к `context`, `update`, `gspread`, глобальному кэшу. Все зависимости — через параметры.

### 3.2 Тесты

Настроить pytest:
- Добавить `pytest` в `requirements-dev.txt` (новый файл — рантайм не засоряем).
- Создать `tests/` рядом с `core/`.
- `pyproject.toml` или `pytest.ini` с `testpaths = ["tests"]`, `pythonpath = ["."]`.

Написать тесты (минимум указан, можно больше):

**`tests/test_dates.py`** — не меньше 8 кейсов:
- `05.04` при сегодня `2026-04-22` → `2026-04-05`;
- `30.12` при сегодня `2026-01-15` → `2025-12-30` (зимняя дата как прошлый год);
- `23.04` при сегодня `2026-04-22` → ошибка «дата в будущем»;
- `01.04` при сегодня `2027-04-22` → ошибка «старше года»;
- некорректный формат (`2026-04-05`, `5/4`, `пять апреля`) → ошибка с понятным сообщением;
- граничные случаи `29.02` в високосный и не-високосный год.

**`tests/test_water.py`** — не меньше 6 кейсов:
- `"2,5"` → `2.5`;
- `"2.5"` → `2.5`;
- `"0,4"` → `0.4`;
- `"25"` — принимается как `25.0` (допустимо) ИЛИ ошибка — уточнить по текущему поведению и зафиксировать в тесте;
- пустая строка, нечисловой мусор → ошибка;
- форматирование обратно: `2.5` → `"2,5"`, `3.0` → `"3"` (или `"3,0"` — как сейчас в карточках).

**`tests/test_service_rules.py`** — не меньше 6 кейсов:
- точка обслужена вчера → «можно не обслуживать сегодня»;
- обслужена позавчера → «нужно сегодня»;
- обслужена сегодня → «уже обслужены»;
- ни разу не обслуживалась → «нужно сегодня»;
- критический сигнал (`запаса для пополнения нет` + `не хватит до следующего приезда`) → попадает в «срочно пополнить» независимо от даты;
- точка «Дома» в обслуживании не участвует (только в ревизии).

**`tests/test_revision_import.py`** — не меньше 8 кейсов:
- блочный формат с одной точкой;
- блочный формат с несколькими точками;
- табличный формат через `/`;
- `Кофе исп` + `Кофе кит` складываются в `Кофе`;
- `Салфетки сухие` распознаётся;
- конфликт значений по одному товару на одной точке → предупреждение + последнее значение;
- пустой ввод → валидная ошибка;
- ввод только с заголовками без данных → валидная ошибка.

**`tests/test_rows.py`** — не меньше 4 кейсов:
- `build_service_row_values` формирует правильный порядок полей, соответствующий заголовку листа;
- `serialize_salary_workers` / `normalize_salary_workers` — симметричные, пустой список, один работник, несколько работников, разделитель с пробелами.

### 3.3 CI и проверки

- Обновить секцию «Проверки» в README.

## Фаза 4. Разбиение монолита

**Зачем.** 92 состояния в одном `ConversationHandler`, 12 700 строк, 0 классов. Любое изменение в ветке диалога рискует зацепить соседние. Запускать эту фазу только после Фазы 3 — тесты служат страховкой.

### 4.1 Целевая структура

```
coffee-bot/
  bot.py                 # точка входа, сборка Application и корневого ConversationHandler
  core/                  # pure-функции (из Фазы 3)
  sheets/                # работа с Google Sheets
    __init__.py
    client.py            # get_sheet, _BOOK_CACHE, лок, retry-декоратор
    services.py          # CRUD по листу обслуживания
    revisions.py
    rent.py
    repairs.py
    travel.py
    salary.py
  handlers/              # Telegram-обработчики по доменам
    __init__.py
    common.py            # start, cancel, главное меню, error_handler
    service.py           # обслуживание точек + нехватка + фото
    revision.py          # ревизия + импорт из текста
    repair.py            # ремонт кофемашин
    rent.py              # аренда
    travel.py            # проезд
    report.py            # отчёт за день / период
    salary.py            # зарплатные задачи
    info.py              # «Информация по точкам»
    group_report.py      # импорт из рабочей группы
  states.py              # enum всех состояний
  keyboards.py           # билдеры InlineKeyboardMarkup (если общие)
  config.py              # переменные окружения, константы POINTS/USERS/SUPPLIES
```

### 4.2 Подход к разбиению

Работать маленькими шагами, каждый — рабочий бот:

1. Вынести константы и настройки в `config.py` (POINTS, SUPPLIES, REVISION_ITEMS, REPAIR_*, env-парсинг). `bot.py` импортирует их обратно. Прогнать тесты и smoke-test.
2. Вынести состояния (`range(92)`) в `states.py` как `IntEnum` либо сохранить текущий стиль, но в отдельном файле.
3. Вынести `sheets/client.py` (кэш, лок, retry-декоратор, `get_sheet`, `get_cached_worksheet`, `get_or_create_worksheet`).
4. По одному домену переносить CRUD-функции Sheets в `sheets/<domain>.py`.
5. По одному домену переносить handlers в `handlers/<domain>.py`. Для каждого домена, если возможно, выделить свой `ConversationHandler` и связать с корневым через `map_to_parent`. Если связка через общее главное меню ломает UX — оставить один корневой `ConversationHandler`, но импортировать состояния и handlers из модулей.
6. В финале `bot.py` — только сборка Application, регистрация handlers, запуск polling. Цель: ≤ 200 строк.

**Важно.**
- Поведение и экраны бота — без изменений.
- Имена callback-паттернов (`sp_`, `svc_ok`, `rev_confirm_save`, `fix_entry_salary_save`, `grp_report_...` и т.д.) — не менять. Persistence через pickle завязан на имена состояний и handlers.
- `PicklePersistence` — проверить, что после разбиения существующий `bot_state.pickle` читается без ошибок. Если требуется миграция — сделать один раз в `on_app_startup` с бэкапом файла.
- На каждом шаге прогонять `pytest`, `ruff`, `py_compile` и ручной smoke-test основных сценариев: обслуживание точки с фото, ревизия за текущий месяц, отчёт за день, создание ремонта с переходами статусов.

### 4.3 Документация

- Обновить `PROJECT_CONTEXT.md`: добавить секцию «Структура кода» с описанием модулей.
- Обновить README: команды запуска, структура, как добавлять нового сотрудника (через Sheets — по итогам Фазы 1).
